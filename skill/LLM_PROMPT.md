# Инструкция для LLM: анализ ASCII-представления изображения

Ты получил файл `output.md` — результат конвертации изображения в текстовый формат, пригодный для анализа языковой моделью.

## Структура файла

Файл содержит несколько секций в формате Markdown. Анализируй их последовательно.

---

### 0. EXIF Metadata (метаданные, опционально)

```markdown
## EXIF Metadata
[метаданные: дата съёмки, камера, разрешение, ориентация и т.д.]
```

- Появляется только если использован флаг `--exif`.
- Содержит техническую информацию: модель камеры, дату съёмки, ISO, выдержку, ориентацию.
- Полезна для понимания контекста фотографий и документов.

---

### 1. OCR Text (распознанный текст)

```markdown
## OCR Text
[текст, извлечённый из изображения]
```

- Содержит все текстовые надписи, обнаруженные на изображении.
- Может содержать ошибки распознавания (особенно для разнонаправленного текста, мелкого шрифта, текста на цветном фоне).
- Используй OCR-текст **как подсказку** — сопоставляй надписи с визуальными элементами из ASCII-арта.

---

### 2. Color Map (карта цветов)

```markdown
## Color Map
Legend:
  R = red    G = green    B = blue    ...
Grid: (20x8)
WWWWWWWWRRRWWWWWW...
```

- **Легенда**: однобуквенные коды и соответствующие им цвета.
- **Сетка**: двумерная матрица, где каждая буква — усреднённый цвет соответствующей области изображения.
- Размер сетки указан в скобках: `(столбцы x строки)`.
- Используй для понимания цветового кодирования диаграммы (например, красный = критический компонент, зелёный = внешняя система, синий = база данных).

---

### 3. ASCII Art (визуальное представление)

Секции вида:
```markdown
<!-- width=100 -->
```
[ASCII-арт]

- Изображение представлено символами, где **каждый символ = 1 блок пикселей**.
- Чем больше `width`, тем детальнее изображение.
- При нескольких ширинах (`--multi 80 150`):
  - Маленькая ширина (80) — **общая форма/силуэт**.
  - Большая ширина (150+) — **детали и текст внутри изображения**.

#### Интерпретация символов

Символы упорядочены от тёмного к светлому:
```
@  %  #  *  +  =  -  :  .  [пробел]
█  ▓  ▒  ░  [пробел]            (если использован --blocks)
тёмный ← → светлый
```

При использовании `--invert` (рекомендовано): **тёмный символ на светлом фоне** — как обычный текст.

При использовании `--binary`: только два уровня — **чёрное и белое**, без полутонов.

#### Как читать ASCII-арт

- **Сплошные горизонтальные линии** (`########`, `=====`) — границы блоков, разделители.
- **Сплошные вертикальные полосы** — боковые границы элементов.
- **Резкие переходы** от `@` к `.` — границы фигур/блоков.
- **Плавные переходы** (`@%#*+=-:.`) — градиенты, тени, скругления.
- **Разрывы в линиях** (`###..###`) — текст внутри блока или вход/выход стрелки.
- **Повторяющиеся паттерны на соседних строках** — протяжённые элементы (боковые панели, колонки).
- **Одиночные компактные группы символов** — иконки, кнопки, мелкий текст.

#### Аннотации

Если в ASCII-строках справа есть текст после `│` — это OCR-подписи, сопоставленные с соответствующими строками изображения. Используй их для идентификации элементов.

## Алгоритм анализа

1. **Прочитай EXIF** (если есть) — получи контекст изображения.
2. **Прочитай OCR-текст** — получи список всех надписей.
3. **Изучи Color Map** — пойми цветовое кодирование.
4. **Начни с самой маленькой ширины ASCII** — определи общую композицию:
   - Где верх/низ/лево/право?
   - Сколько основных блоков/областей?
   - Есть ли стрелки/соединения между блоками?
5. **Перейди к большей ширине** — рассмотри детали каждого блока:
   - Какие надписи (из OCR) к каким блокам относятся?
   - Какие цвета у каких блоков?
   - Как блоки соединены?
6. **Сформулируй ответ**:
   - Что изображено (схема, фото, скриншот, документ)?
   - Перечисли основные элементы и их взаимосвязи.
   - Опиши поток/алгоритм, если это диаграмма.
   - Укажи, что именно не удалось распознать.

## Примеры интерпретации

### Схема интеграции
```
########    ========    ########
# СУБД # -> =шина  = -> # ИС   #
########    ========    ########
```
- Три блока, соединённых стрелками.
- Левый блок: СУБД (OCR: «БД Интегратора»).
- Центр: шина сообщений (OCR: «RabbitMQ»).
- Правый блок: целевая система (OCR: «КИСУСС»).
- Вывод: схема передачи данных от БД через шину в целевую систему.

### Блок-схема алгоритма
```
        ┌──────┐
        │ВЫЗОВ │
        └──┬───┘
      ┌────┴────┐
      │ОПЕРАТОР │
      │свободен?│
      └─┬────┬──┘
    да  │    │  нет
   ┌────┘    └────┐
   ▼              ▼
```
- Ромбы — ветвления (условия).
- Прямоугольники — действия.
- Стрелки — переходы.
- Вывод: алгоритм обработки вызова с проверкой наличия оператора.

## 5. Выходной формат для диаграмм и схем

Если изображение является **диаграммой, схемой, блок-схемой или архитектурой**,
сохрани файл расшифровки `{имя}_scheme.md` **в ту же папку**, где находится `output.md`.
Файл должен содержать:

### Обязательные секции
1. **Общее описание** — 1-2 предложения о назначении схемы
2. **Mermaid-диаграмма** — визуальный граф в формате Mermaid:
   - `flowchart TB` для вертикальных схем, `flowchart LR` для горизонтальных
   - Группировка в `subgraph` по слоям (телефония, backend, данные и т.д.)
   - Ромбы `{текст}` для ветвлений/условий
   - Пунктир `-.->` для неактивных/отключённых связей
   - ❌ для неиспользуемых компонентов
3. **Компоненты/слои** — таблица: компонент → назначение
4. **Потоки данных/связи** — описание маршрутов между компонентами
5. **Привязка к строкам ASCII** — таблица: диапазоны строк → содержание

### Пример структуры файла

```markdown
# Название схемы — имя_файла.png

## Общее описание
[суть схемы в 1-2 предложения]

## Mermaid-диаграмма
`​``mermaid
flowchart TB
    subgraph layer1 [Название слоя]
        A[Компонент]
        B[Компонент]
    end
    A --> B
`​``

## Компоненты
| Компонент | Статус | Назначение |
|-----------|--------|-----------|
| ...       | ✅/❌   | ...       |

## Потоки данных
1. **Маршрут**: A → B → C

## Привязка к строкам ASCII
| Строки | Содержание |
|--------|-----------|
| 1-10   | ...       |
```

Важно: **исправляй ошибки OCR** в названиях компонентов, ориентируясь на контекст (например, «оператор» вместо «оператдр», «RabbitMQ» вместо «RabbitiQ»).

---

## Важно

- Не домысливай детали, которых нет в ASCII-арте или OCR.
- Если текст не читается — отметь это явно.
- Если соединения между блоками не прослеживаются — укажи неуверенность.
- Для фотографий (без OCR и с плавными градиентами) — опиши общую сцену, объекты, их взаимное расположение, но не утверждай детали, неразличимые в ASCII.
